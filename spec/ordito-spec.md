# Ordito 仕様書（ドラフト v0.4）

> **Ordito** — AIがドキュメントを生成・更新する、オープンな文書生成システムの規格。
> 名は織物の経糸（ordito / orditura）に由来する。構造（IR）という経糸を先に張り、
> その上にAIが本文を織り上げる、という設計思想を体現する。

---

## 0. この文書について

本仕様は、AIによってドキュメントを生成・更新するシステム **Ordito** の設計を規定する。
目的は、特定の実装に縛られない「規格」を定めることであり、誰でもこの規格に準拠した
テンプレート・生成エンジン・入出力プラグイン（スキル）を独立に実装できることを目指す。

Ordito はオープンソースとして公開され、規格本体・参照実装・一通りのワークフローを
揃えた形で GitHub 上に置かれることを想定する。

本文書は2つの層に分かれる。

- **規格（normative）**: 準拠する実装が必ず守るべき契約。中間表現とスキルの入出力を定める。
- **参照実装（informative）**: 規格を満たす具体的な作り方の一例。差し替え可能。

キーワード「しなければならない（MUST）」「すべき（SHOULD）」「してよい（MAY）」は、
準拠判定における要件の強さを示す。

ステータス: ドラフト。POC を二周完了済み（第1弾: 単一IR→生成→単一HTML、第2弾: 複数ページ＋
コレクション＋内部リンク＋混在生成）。本v0.4は第2弾の所見（`findings-poc2.md`）を反映したもの。
`TBD` は未決事項を示す。

### v0.4 での主な改訂（POC第2弾由来）

- **コレクションの詳細スキーマ確定**（3.5）: 多階層グループ・`label` 上書き・`order` 規則を規定。
- **物理パスは階層保持に確定**（3.6）: doc id の階層を出力パスに保つ。フラット方式は slug 衝突のため不採用。
- **枠内リンクの相対解決を義務化**（4.7）: 「枠は固定」でも枠内のリンク（ホーム・ロゴ等）はページ深さに
  依存するため、生成エンジンが現在ページ基準で相対解決する MUST（POCでブランドリンク切れを実測）。
- **ブロックの JSON Schema を準拠テストとして同梱**（3.3・9章）: 語彙を機械可読スキーマ化。
- **`field_map` の未マップ検出を動的化＋`OMIT` 慣習**（4.4）: 既知の固定リストではなくブロックの実キーから
  導出し、未知フィールドの脱落も検出。「表示しない」は `"OMIT"` で明示。
- **`params.default` の型を拡張**（3.3）: string/number/boolean/null。看板例 `expires_in=3600`（整数）を表現可能に。
- **入れ子ブロックのレベルはコンテナ追従に確定**（5.2）: 将来「個別指定」を足せる余地は残す。
- **孤立ページの扱いを規定**（3.7）: 参照先欠落は警告スキップ、ナビ未掲載は生成対象外（限定生成の余地は残す）。

### v0.3 での主な改訂（第1弾POC由来）

- **フィールドマッピングの網羅義務**（4.4）: 契約がブロック型の各フィールドの行き先を網羅
  定義しないと、IRの情報が静かに失われる（POCで `params.default` の沈黙脱落を実測）。これを規格で禁じた。
- **サイト構造レイヤの新設**（3.5）: ナビ項目はドキュメント集合に依存し、単一IRでは決まらない。
  サイトマップ（コレクション）という上位概念を追加。
- **テンプレート契約スキーマの拡張**（4.2）: `render_hint` / `variant_class` / 許可リスト群を正式化。
  AIへの渡し方は許可リスト型（schema戦略）を既定とする（4戦略の実測比較に基づく）。
- **インライン記法の規格化**（3.3.1）: `paragraph.text` の軽量マークアップを規格で定義。
- **検証の二層化**（6章）: 機械チェック（決定論）＋意味チェック（IR忠実度）の二段に明確化。

### データ交換形式の原則

Ordito 内部のすべてのやり取り（IR、テンプレート契約、スキル間の受け渡し）は
**JSON で統一する MUST**。これにより、スキル・エンジン・テンプレートが言語非依存で
連携でき、検証も機械的に行える。

> POC所見: 能力の高いモデルは、契約がJSONで機械可読であること自体によって、薄い足場でも
> 契約を遵守できた。JSON統一の原則はAI生成との相性の面でも有効と確認された。

## 1. 背景と狙い

従来の静的サイトジェネレータ（Docusaurus など）は、コンテンツ・表示・ビルドが密に結合している。
Ordito はこれらを分離し、次の構図を取る。

- **中身（コンテンツ）はAIが扱う構造化データ（IR）として持つ。**
- **見た目（枠）はテンプレートが固定で持つ。**
- **両者を合成して、AIが最終的なHTML/CSSを生成する。**

これにより、入力・出力・既存ドキュメントからの移行を、すべて同じ契約の上に乗せられる。
従来ツールは「移行元」としてのみ関わり、Ordito の実行時には関与しない。

### 設計思想

- **制約された多様性（constrained variation）**: ページごとに本文や構造が変わるのは当然として、
  デザインの揺らぎも積極的に許容する。ただし無制限ではない。デザイントークン
  （配色・タイポグラフィ・余白）と全体レイアウト（ナビ・ヘッダー）はテンプレートが固定し、
  その内側でAIが自由に組む。「許容するばらつき」と「許容しないばらつき」を明確に分ける。
  - **重要な境界（POC由来）**: デザインや内側HTML構造の揺らぎは許容するが、**IRの情報の欠落は
    許容しない**。前者は「制約された多様性」、後者は「データ欠落」であり、別物として扱う（4.4）。
- **過渡期を楽しむ**: 現在のAIは発展途上であり、出力の揺らぎはバグではなく時代の特徴と捉える。
  仕様はこの揺らぎを排除せず、安全な範囲に閉じ込める方針を取る。
- **揺らぎは制御可能（POC由来）**: 契約の `render_hint` の詳細度が、許容する揺らぎの量を直接決める。
  曖昧に書けば多様に、全フィールドまで具体化すれば一貫する。揺らぎは思想であると同時に操作可能なパラメータ（4.3）。
- **データ更新と生成の分離**: IR（元データ）の更新と、成果物（HTML）の生成は別工程とする。
  更新は軽く頻繁に、生成は明示的なタイミングでまとめて。詳細は5章・7章。

## 2. 全体アーキテクチャ

```
[移行元: Docusaurus等] --(移行スキル)--> [IRストア] <--(取り込み/更新スキル)-- [新規・更新]
                                            |
                                  本文(IR)＋メタ＋サイトマップ (JSON)
                                            |
                                  (明示トリガーで生成)
                                            |
                                            v
                                   [AI生成エンジン] <---- [テンプレート（枠＋契約, JSON）]
                                            |
                                            v
                                 [成果物: HTML/CSS ページ]
                                            |
                                     (出力スキル)
                                            |
                                 [公開サイト / PDF / 配信API]
```

中心は **IRストア**。これが単一の真実の源（single source of truth）であり、
すべてのスキルはこのストアに対して「書く」か「読む」かのいずれかを行う。

重要な点として、ストアへの書き込み（更新）は、成果物の生成を自動的には引き起こさない。
生成は独立した明示トリガーで走る（5.4 参照）。

## 3. 中間表現（Intermediate Representation, IR）

### 3.1 役割と形式

IR は、表示形式から独立した「本文の内容そのもの」を保持する。
IR は HTML を**含まない**。見た目に関する情報を持たない。
何が書かれているか（構造と意味）だけを持つ。

物理フォーマットは **JSON とする MUST**（データ交換形式の原則に従う）。
人手での編集や Git 差分の読みやすさを補助するため、参照実装は
IR JSON ⇔ 人間可読形式（フロントマター付き Markdown 等）の相互変換ツールを
提供してよい MAY。ただし正本は JSON とする。

### 3.2 ドキュメントの構造（規格）

各ドキュメントは1つの IR ドキュメントとして表現される。最小構成は次のとおり。

```json
{
  "id": "guides/authentication",
  "meta": {
    "title": "認証ガイド",
    "order": 3,
    "tags": ["auth", "getting-started"],
    "updated_at": "2026-06-13T00:00:00Z",
    "generated_at": null,
    "sources": [{ "type": "migration", "ref": "docusaurus/auth.md" }]
  },
  "blocks": [
    { "id": "b1", "type": "heading", "level": 1, "text": "認証ガイド" },
    { "id": "b2", "type": "paragraph", "text": "API キーは `Authorization` ヘッダで送る。詳細は [認証](guides/authentication) を参照。" },
    { "id": "b3", "type": "code", "lang": "bash", "text": "curl -H 'Authorization: Bearer ...'" },
    { "id": "b4", "type": "params", "items": [ { "name": "api_key", "type": "string", "required": true, "desc": "...", "default": null } ] },
    { "id": "b5", "type": "note", "variant": "warning", "text": "..." }
  ]
}
```

- `id`（MUST）: ドキュメントの一意な識別子。論理パスとして扱う（3.5・物理パスへの解決は生成時）。
- `meta`（MUST）: タイトル・並び順・更新時刻・出典を持つ。
  - `updated_at`: IR が最後に更新された時刻。
  - `generated_at`: 成果物が最後に生成された時刻。`updated_at` より古ければ「未反映」と判定できる（5.4）。
  - `sources`: その記述がどこ由来か（移行・API取り込み・手動など）を記録し、検証と追跡を可能にする。
- `blocks`（MUST）: 本文を意味ブロックの配列として持つ。各ブロックは `id` と `type` を持つ MUST。
  - ブロック `id` は差分更新の単位となる（3.4）。

### 3.3 ブロックの種類（語彙・確定）

ブロックの `type` の集合が、Ordito の「語彙」である。
開発者ドキュメントは要素の種類が限られるため、次の語彙で大半をカバーできる。
各フィールドのスキーマは POC を経て次のとおり確定する（旧版では TBD だった部分）。

| type | 意味 | フィールド（確定） |
|------|------|--------------------|
| `heading` | 見出し | `level`(1-6, 描画時 h1-h4 にクランプ), `text` |
| `paragraph` | 本文段落 | `text`（インライン記法を許容, 3.3.1） |
| `code` | コードブロック | `lang`, `text`, `filename?` |
| `params` | パラメータ表 | `items[]`（`name`, `type`, `required`, `desc`, `default?` ＝ string/number/boolean/null） |
| `table` | 一般の表 | `headers[]`, `rows[][]` |
| `note` | 注記・警告 | `variant`(info/warning/danger), `text` |
| `list` | 箇条書き / 番号付き | `ordered`(bool), `items[]` |
| `link` | リンク（ブロック） | `href`(3.3.2), `text` |
| `tabs` | タブ切り替え | `tabs[]`（`label`, `blocks[]` ＝**ブロックの入れ子を許容**, 3.3.3） |
| `steps` | 手順 | `items[]`（`{ title?, text }` または文字列） |

> **`inline_code` の語彙からの削除（v0.3）**: 旧版の独立ブロック `inline_code` は、主用途が段落内であり
> 独立ブロックとしての用途が薄いため、語彙から外し `paragraph` のインライン記法（3.3.1）に統合した。
>
> 語彙は拡張可能（MAY）。準拠実装は未知の `type` に出会ったとき、無視せず明示的にエラーまたは
> フォールバック描画する SHOULD（参照実装は `note(warning)` にフォールバック）。

> **JSON Schema 化（v0.4）**: 上記の語彙は機械可読な JSON Schema（draft 2020-12）として書き起こし、
> 準拠テスト（9章）の一部として同梱する。各ブロック型を `$defs/blocks/<type>` に定義し、`type` を `const`、
> `additionalProperties: false` とすることで、`$defs/block` の `oneOf` が常に1分岐のみ一致する
> （標準語彙だけでディスクリミネータと同等の判別が成立する）。
>
> **`params.default` の型拡張（v0.4）**: 第1弾の `default` 沈黙脱落を受けて 4.4 を設けたが、その看板例
> `expires_in=3600`（整数）を、当初の `string|null` 限定スキーマでは表現できなかった（第2弾レビューで判明）。
> `default` は **string / number / boolean / null** を許容する。`null` は「既定なし」を明示的に表す。

#### 3.3.1 インライン記法（規格）

`paragraph.text` 等のインライン要素は、IR が HTML を含まない原則（3.1）に従い、
**規格化された軽量マークアップ**で持つ MUST。HTML を直接埋め込んではならない MUST NOT。
初期に定める記法（POCで採用・確定）:

- インラインコード: `` `code` ``
- リンク: `[label](href)`（`href` は 3.3.2 の解決規則に従う）
- 強調: `**bold**`

> 記法の拡張は MAY だが、規格に追加されるまでは上記のみを正規とする。
> 生成時にこの軽量マークアップを、テンプレート契約が許可するインライン用クラス／タグへ変換する。

#### 3.3.2 リンクの参照規則（規格）

`link.href` および インラインリンクの `href` は、次の二種を区別する MUST。

- **内部参照**: 既知のドキュメント `id`（例 `guides/authentication`）。生成時に物理パスへ解決する。
- **外部参照**: スキーム付き URL（`https://...` 等）。素通しする。

IR は表示非依存であるため、内部リンクは**論理参照（doc id）**で保持し、HTMLパスへの解決は
**生成時**に行う MUST。これにより出力構成（ファイル名規則・階層）を変えてもIRは不変に保てる。

#### 3.3.3 ブロックの入れ子（規格）

`tabs` 等、子ブロックを持つブロック型は、`blocks[]` として 3.3 の語彙を**再帰的に**含められる MUST。
入れ子の各ブロックも `id` を持つ MUST（差分更新の単位を保つため、3.4）。

### 3.4 差分更新の単位

差分更新の基本単位は **ブロック（block.id）** とする。
更新スキルは、変わったブロックだけを書き換え、`meta.updated_at` を更新する MUST。
成果物の再生成時には、どのページ（IRドキュメント）が前回生成以降に更新されたかを
`updated_at` と `generated_at` の比較で判定し、該当ページのみ作り直せる SHOULD。

### 3.5 サイト構造（コレクション）— v0.3 で新設・v0.4 で詳細確定

単一の IR ドキュメントは「1ページの中身」しか表さない。一方、ナビゲーション項目・ページの並び・
階層といった**サイト全体の構造**は、ドキュメントの集合に依存し、単一 IR からは決まらない。
この層を表す上位概念として **コレクション（サイトマップ）** を規格に追加する。

```json
{
  "collection_id": "dev-docs",
  "title": "開発者ドキュメント",
  "nav": [
    { "doc": "guides/quickstart", "order": 1 },
    { "group": "ガイド", "order": 2, "items": [
      { "doc": "guides/authentication", "order": 1 },
      { "doc": "guides/errors", "order": 2, "label": "エラー一覧" }
    ]},
    { "group": "API リファレンス", "order": 3, "items": [
      { "doc": "reference/endpoints", "order": 1 },
      { "group": "詳細", "order": 2, "items": [
        { "doc": "reference/pagination", "order": 1 }
      ]}
    ]}
  ]
}
```

`nav[]` の各項目は次の2種のいずれか（`oneOf`）:

- **ドキュメント参照**: `{ doc, order?, label? }`。`doc` は IR の `id`。`label` はナビ表示名の上書き
  （省略時は IR の `meta.title`）。
- **グループ**: `{ group, order?, items[] }`。`items` は `navItem` を**再帰的**に含められる
  ＝**多階層グループ**を表現する（POCで「API リファレンス > 詳細 > ページネーション」を実証）。

規定（POCで確定）:

- **並び順**: 各階層内で `order` 昇順、`order` 省略時は配列の出現順（安定ソート）。`order` を任意にしたのは
  手書きで並べやすくするため。
- **所有権の分離**:
  - ナビの**項目・順序・グループ**は **コレクション**が所有する（本節）。
  - ナビの**描画構造とスタイル**は **テンプレート**が所有する（4章）。
  - 生成エンジンは両者を合成し、現在ページに `aria-current` を付与し、リンクを相対化する（3.6・4.7）。
- **`meta.order` との関係**: 単一 IR の `meta.order` はヒントに留め、**最終的な並びはコレクションの `order` が決定する**
  （所有分離の徹底）。

> 未対応（TBD）: 複数コレクション間の関係、ナビ項目への外部リンク配置、`doc` を伴わない見出しのみの項目。

### 3.6 出力の物理パス規則 — v0.4 で確定

IR の `id` は論理パスであり、出力ファイルの物理パスへは**生成時に解決する**（3.1・3.3.2）。
解決は **doc id の階層を保持する** MUST。

- `guides/quickstart` → `guides/quickstart.html`、`reference/pagination` → `reference/pagination.html`。
- **フラット方式（末尾セグメントのみ）は採らない**。`guides/quickstart` と `reference/quickstart` のように
  別階層で末尾が同名になると **slug 衝突**するため。階層保持なら衝突しない。相対パス計算の手間は
  生成エンジンが吸収する。
- 予約 id: `index` はサイトのホームと衝突するため、ドキュメント id として使ってはならない MUST NOT。

> 参照実装は、コレクション無しの単独生成（フラットモード）に限り、後方互換として末尾セグメント＋
> ルート直下出力を維持してよい MAY。

### 3.7 孤立ページの扱い — v0.4 で規定

コレクション（3.5）と IR ドキュメントは独立に存在するため、両者がずれると2種の孤立が生じる。
意味が異なるため、扱いを分ける。

- **参照先の欠落**（コレクションが参照する `doc` に対応する IR が無い）: 読者がナビから辿ると行き先が無い
  実害がある。生成エンジンは**警告して当該項目をスキップする** SHOULD。
- **ナビ未掲載**（IR は存在するがコレクションのどこにも無い）: 「下書き」「直リンク限定ページ」など
  意図的な運用の場合がある。既定では**生成対象外**とする SHOULD。

> 将来の余地（TBD）: ナビ未掲載ページを「明示フラグで限定生成（ナビには出さないが生成はする）」できる
> オプションを設けてよい MAY。現状は既定挙動のみを規定する。

## 4. テンプレート（枠）

### 4.1 役割

テンプレートは「枠」を固定で提供する。AIはこの枠に手を出さない。

テンプレートが持つもの（MUST）:

- **全体レイアウト**: ヘッダー、フッター、サイドバー、本文の差し込み口、およびナビの**描画構造とスタイル**
  （ナビの**項目**はコレクションが所有, 3.5）。
- **デザイントークン**: 配色、タイポグラフィ、余白、角丸などの基本値。
- **コンポーネントライブラリ**: 3.3 の各ブロック型に対応する、定義済みの描画要素（CSSクラス／コンポーネント）。
- **枠内リンク**: ホーム・ロゴ等、枠が持つリンクの**差し込み口**（実体のパスは生成時に相対解決, 4.7）。

### 4.2 テンプレート契約（JSON）— v0.3 で拡張

AIおよび検証器に渡される「契約」は **JSON で表現する MUST**。POC の結果、v0.2 の最小例
（`template_id` / `components` / `content_slot` / `constraints`）では機械的に使うには情報が不足したため、
次のフィールドを正式化する。

```json
{
  "template_id": "dev-docs-standard",
  "content_slot": "#doc-body",
  "components": [
    {
      "for": "heading",
      "class": "doc-h",
      "render_hint": "<h{level} class=\"doc-h\">{text}</h{level}>",
      "field_map": { "level": "タグ h1-h4 に対応", "text": "見出し本文" }
    },
    {
      "for": "code",
      "class": "doc-code",
      "render_hint": "<div class=\"doc-code\" data-lang=\"{lang}\"><pre><code>{text}</code></pre></div>",
      "sub_classes": ["doc-code__lang"],
      "field_map": { "lang": "data-lang属性", "text": "code内テキスト", "filename": "doc-code__name に表示" }
    },
    {
      "for": "note",
      "class": "doc-note",
      "variant_class": "doc-note--{variant}",
      "variants": ["info", "warning", "danger"],
      "render_hint": "<div class=\"doc-note doc-note--{variant}\">{text}</div>",
      "field_map": { "variant": "variant_class に展開", "text": "本文" }
    },
    {
      "for": "params",
      "class": "doc-params",
      "sub_classes": ["doc-params__name", "doc-params__type", "doc-params__req", "doc-params__default", "doc-params__desc"],
      "render_hint": "<table class=\"doc-params\"><thead>...</thead><tbody><tr><td class=\"doc-params__name\">{name}</td><td class=\"doc-params__type\">{type}</td><td class=\"doc-params__req\">{required}</td><td class=\"doc-params__default\">{default}</td><td class=\"doc-params__desc\">{desc}</td></tr></tbody></table>",
      "field_map": { "name": "...", "type": "...", "required": "...", "default": "doc-params__default 列（null は「—」）", "desc": "..." }
    }
  ],
  "allowed_html_tags": ["h1","h2","h3","h4","p","a","code","pre","div","span","table","thead","tbody","tr","th","td","ul","ol","li","strong"],
  "allowed_classes": ["doc-h","doc-code","doc-code__lang","doc-code__name","doc-note","doc-note--info","doc-note--warning","doc-note--danger","doc-params","doc-params__name","doc-params__type","doc-params__req","doc-params__default","doc-params__desc","doc-ic"],
  "allowed_attributes": ["class","href","data-lang"],
  "constraints": {
    "allow_custom_css": false,
    "allow_layout_edit": false,
    "must_render_all_blocks": true,
    "must_map_all_fields": true
  }
}
```

各フィールドの意味と、なぜ必要か（POC由来）:

- **`render_hint`**（SHOULD・遵守度を最も左右）: 期待 HTML 構造を文字列例で示す。クラス名の列挙だけでは
  `<div class="doc-code"><pre><code>` のような**入れ子構造**を AI が一意に再現できない。構造例があると安定する。
  詳細度が高いほど揺らぎが減る（4.3）。
- **`variant_class`**（variant を持つ型は MUST）: `note` 等の variant をクラスへ展開する規則
  （`doc-note--{variant}`）。無いと AI が `doc-note-warn` 等の独自命名をしがち。
- **`field_map`**（MUST, 4.4）: ブロック型の**各フィールドの行き先**を網羅定義する。未マップのフィールドは
  情報欠落の原因になる。
- **`sub_classes`**: コンポーネント内部の子要素クラス。許可リストに含めるために必要。
- **`allowed_classes` / `allowed_html_tags` / `allowed_attributes`**（MUST）: 検証器が機械照合するための
  ホワイトリスト。契約は**許可リスト（肯定形）**で表現する——禁止列挙だけだと検証器が「何が許されるか」を
  別途持つ二重管理になる。`style`/`id` 禁止も「許可リストに無い」として肯定的に表現できる。
- **`constraints`**: 機械可読フラグ。`must_render_all_blocks`（全ブロック描画）/ `must_map_all_fields`
  （全フィールド対応, 4.4）を含む。

AIがしてはならないこと（MUST NOT）:

- 新しいデザイントークンの発明（独自の色・フォント・余白の指定）。
- ナビ・レイアウトなど枠領域の改変。
- `content_slot` の**外側**への出力（AI 出力は `content_slot` の **innerHTML（内側フラグメント）**である MUST, 5.1）。

### 4.3 揺らぎの制御（render_hint の詳細度）— POC由来

POC の比較で、`render_hint` を名前セルしか示さず残りを省略した場合、AI と参照実装が
**それぞれ妥当だが異なる**内側構造を選んだ（どちらも許可クラス・許可タグの範囲内で、描画も崩れない）。
これは「制約された多様性」が機能している実例であると同時に、次を意味する。

> **`render_hint` の詳細度が、許容する揺らぎの量を直接コントロールする。**

- 揺らぎを許したい（過渡期を楽しむ）箇所は、`render_hint` を要点だけにとどめる。
- 一貫性を高めたい箇所は、`render_hint` を全フィールドの構造まで具体化する SHOULD。

これにより「制約された多様性」は抽象方針ではなく、契約上の操作可能なツマミになる。

### 4.4 フィールドマッピングの網羅義務 — 最重要規定（第1弾POC由来・第2弾で強化）

POC で、IR の `params` に存在した `default`（例 `expires_in=3600`）が、AI 出力で**静かに脱落**した。
原因は、契約の `params` コンポーネントが `default` の行き先を定義していなかったこと。AI は与えられた
`render_hint` に忠実だった結果、IR にある情報を落とした。これは検証の客観チェック（クラス遵守）を
**通過してしまう**——クラスは守られているからである。

この事象は「制約された多様性」の許容範囲では**なく**、**データ欠落**である（1章の境界を参照）。
規格として次を定める。

- 契約は、対象ブロック型の**各フィールドの行き先**を `field_map` で網羅的に定義する MUST。
- 生成エンジン／検証器は、IR ブロックに存在するが契約で**未マップのフィールド**を検出した場合、
  警告する SHOULD（POC の客観バリデータはこれを見逃したため、意味チェック層が必要・6章）。
- 表示しないと決めたフィールドも、**`"OMIT"` を値として明示的にマップする** MUST（暗黙の脱落を禁ずる）。
  `field_map` にキーが在れば「マップ済み」、`"OMIT"` はその上で「意図的に非表示」を表す。キーが**無い**ものだけが
  `unmapped_field` 警告の対象。

**未マップ検出は動的に行う（第2弾で強化）**: 検出対象のフィールド集合を、固定リストではなく
**IR ブロックの実キーから導出する** SHOULD（`params`/`steps`/`tabs` 等は item のキーの和集合を取る）。
固定リスト駆動だと、語彙に後から増えた・想定外の未知フィールド（例 `deprecated`）の脱落を取りこぼす。
理想は、ブロックの JSON Schema（3.3）からフィールド集合を引いて、検出と語彙定義を単一ソースに揃えること（TBD）。

> `field_map` の値は現状フリーテキスト（行き先の説明）で、`"OMIT"` のみ予約語。将来は
> `{ target: "<class/attr>", omit?: true }` への構造化を検討してよい（TBD）。

### 4.5 テンプレートとAIの分担

```
テンプレートが固定:  レイアウト / デザイントークン / コンポーネントのCSS / ナビの描画構造
コレクションが固定:  ナビの項目・順序・グループ（3.5）
AIが決める:          どのコンポーネントを使うか / 本文の構成 / コンポーネントへの内容の割り当て
```

これにより、トークンコストが下がる（毎回デザインを指示しないため）と同時に、
ページ間のデザイン一貫性が保たれる。AIの自由は「内容と構造の選択」に集中する。

### 4.6 デフォルトテンプレート

複数のデフォルトテンプレートを同梱する SHOULD（例: 標準的なAPIリファレンス向け、
チュートリアル向けなど）。利用者は選択でき、独自テンプレートを追加してよい（MAY）。

### 4.7 枠内リンクの相対解決 — v0.4 で新設（POC由来）

「枠は固定」（4.1）は描画**構造**の話であり、枠が持つ**リンク**（ホーム・ロゴ・親ページ等）の
**実体パス**は別問題である。POC で、ヘッダーのブランドリンクを `index.html` 固定にしたところ、
サブディレクトリのページ（例 `guides/quickstart.html`）から解決できずリンク切れになった。

規格として次を定める。

- テンプレート枠内のリンクは、**差し込み口（プレースホルダ）**として持つ MUST（実体のパスを直書きしない）。
- 生成エンジンは、各ページの出力位置（3.6）を基準に、枠内リンクを**現在ページからの相対パスへ解決する** MUST。

> 教訓: 「枠が固定」でも、枠内リンクはページ深さに依存する。構造の固定とパスの解決は別レイヤである。
> これは本文中の内部リンク解決（3.3.2）と同じ原理を、枠にも適用するということ。

## 5. AI生成エンジン

### 5.1 入力と出力

- 入力（MUST）: ① IR ドキュメント（JSON）、② テンプレート契約（JSON）、③ コレクション（ナビ生成用, 3.5）。
- 出力（MUST）: `content_slot` の **innerHTML（内側フラグメント）**。slot 要素自体やテンプレートの枠は含まない。
  枠との合成はエンジンが行う。

### 5.2 生成のレベル（実装が選択）

3.4 の粒度に応じ、ブロックごとに次のいずれかで生成する。

1. **構造流し込み**: 構造化ブロック（params, table, steps 等）を、対応コンポーネントに
   機械的に割り当てる（揺らぎ最小・決定論）。
2. **コンテンツエリア生成**: 本文領域のマークアップをAIが組む（語彙の範囲内）。
3. **自由生成**: 枠の外側以外はAIに委ねる（揺らぎ最大、過渡期の実験用、MAY）。

> 参照実装は 1 と 2 を基準線とする SHOULD。本番では params/table/steps を決定論（レベル1）、
> 散文（paragraph 等）を AI（レベル2）に振り分ける混在が妥当（POC所見）。

**入れ子ブロックのレベル（v0.4 で確定）**: `tabs` 等の入れ子ブロックは、レベルを**トップレベルのブロック単位**で
割り当て、**入れ子の子ブロックはコンテナのレベルに従う**（コンテナ追従）。コンテナが決定論（L1）なら
子も決定論で描画する。

- 理由: `tabs` の主用途は言語別コード例（`code`）であり、`code` はもともと L1（決定論）が正しい。
  入れ子の中まで種類別に個別判定する複雑さに見合う出番が、現状では乏しい。
- 将来の余地（TBD）: 入れ子の子ブロックごとにレベルを**個別指定**できる拡張を足してよい MAY。
  コンテナ追従を既定とし、必要が示されてから個別指定を追加する（後方互換な拡張）。

### 5.3 AIへの契約の渡し方（参照実装の既定）— POC由来

「契約をどう渡すと遵守度が高いか」を4戦略で実測した（`rules` / `schema` / `example` / `minimal`）。
結論として、**許可リスト型（schema 戦略：使ってよいコンポーネント表を主役にする）を既定とする** SHOULD。

- 能力の高いモデルでは、契約が JSON で機械可読であること自体が効き、薄い足場でも遵守できた。
- その上で、`schema` 戦略はプロンプト長が `rules` の約半分で遵守度は同等以上＝**コスト効率が最良**。
- 遵守度を最も左右したのは戦略名よりも `render_hint`（期待構造の例示）の有無・詳細度だった（4.3）。

> これは参照実装の推奨であり、規格上の MUST ではない（MAY で他戦略を採ってよい）。

**混在生成時、AI には論理参照のまま返させる（v0.4・POC由来）**: 散文だけを AI に渡して `{id: html}` の形で
返させ、本文順にエンジンが差し込む場合、リンクは **doc id（論理参照）のまま返させ、物理パスへの解決は
エンジンが行う**（3.3.2 の分離を AI 経路でも保つ）。プロンプトで「href を変換するな（`.html` を付けるな）」と
**明示する必要がある** SHOULD。AI は親切心で物理パスに変換しがちで、明示しないと「IR=論理／生成=物理」の
分離が崩れる。

### 5.4 生成は明示トリガー（規格）

生成（IR → 成果物）は、ストアへの書き込みとは独立した工程とする MUST。
書き込みは生成を自動的に引き起こさない。理由は、ビルドのコスト・所要時間、および
AI利用時のトークン／コンテキスト消費を、更新のたびに払いたくないため。

これにより次のワークフローが成立する。

- 軽い更新（スキル経由のIR書き換え）は頻繁・低コストで行える。
- 重い生成は、まとめて・任意のタイミングで・該当ページのみ実行できる（3.4 の判定を利用）。

## 6. 検証（バリデーション）— v0.3 で二層化

生成後、成果物は検証工程を通る MUST。検証は二つの役割を併せ持つ:
**公開前ゲート**であり、かつ **AI生成の品質計測器**でもある（POCで後者の有用性を確認）。

検証は二層で構成する SHOULD。

### 6.1 機械チェック（決定論・安価・MUST）

- 整形式（タグの閉じ忘れ等がない）。
- 許可クラス／許可タグ／許可属性のみ使用（4.2 のホワイトリストで照合）。枠侵食がない。
- `content_slot` の外へ出力していない。
- **未マップフィールドの検出**（4.4, SHOULD）。

> POC教訓: 機械チェックは「クラスを守っているか」は測れるが、`default` の沈黙脱落のような
> **情報の欠落は見逃す**。よって意味チェック層が要る。

### 6.2 意味チェック（IR忠実度・SHOULD）

- IR の全ブロック・全フィールドが出力に反映されているか（欠落・捏造・順序入れ替えの検査）。
- 参照実装では LLM ジャッジ（adversarial）で実施したが、手段は問わない（差分照合等でもよい MAY）。

検証に失敗した成果物は公開してはならない MUST。再生成または差し戻しを行う。
実装は自作・既存ライブラリのいずれでもよい（MAY）。

## 7. スキル / プラグイン

すべての拡張は、ストアに対する **書き込み** か **読み出し** のいずれかである。
スキルは IR / 契約 / コレクションの JSON スキーマだけを介して連携する MUST。

### 7.1 書き込みスキル（→ ストア）

- **移行スキル**: 既存ドキュメント（Docusaurus, 旧Markdown, HTML 等）を IR に変換する。
  初回のみ実行される一回限りの取り込み。出典に `migration` を記録する MUST。
- **取り込みスキル**: API仕様・コード・データソースから IR を生成・更新する。
- **更新スキル**: 知見・変更を受け、該当ブロックの IR を差分更新する。生成は行わない（5.4）。

### 7.2 読み出しスキル（ストア →）

- **生成スキル**: 明示トリガーで IR ＋ テンプレート契約 ＋ コレクションから成果物を生成する。
- **公開スキル**: 成果物を読者向けサイトとして配信する。
- **整形スキル**: IR または成果物を別形式（PDF, OpenAPI, 別システム向けJSON 等）へ変換する。

### 7.3 想定ユースケース（informative）

更新と生成を分離したことで、次のような対話的ワークフローが自然に表現できる。

1. AI利用中に知見が蓄積する。
2. システムが問う:「こういう知見が溜まりました。記載しておきますか？」
   → Yes なら **更新スキル** が IR を書き換える（軽い・低コスト）。
3. 後で別途問う:「更新内容を反映（再生成）しますか？」
   → Yes なら **生成スキル** が該当ページを作り直す（重い・明示的）。

この二段確認は、内部構造（データ更新と生成の分離）がそのままUXに対応した形である。

## 8. 移行（Docusaurus 等からの取り込み）

移行は 7.1 の移行スキルとして実装され、初回の書き込みと同じ経路を通る。
専用の特別経路を設けない（取り込みと移行を同じ契約に乗せる）。

移行スキルの責務:

- 既存ドキュメントを解析し、3.3 の語彙へマッピングする。
- マッピングできない要素は、`note` などへのフォールバック、または手動レビュー待ちとして記録する SHOULD。
- すべてのブロックに出典（元ファイルへの参照）を付与する MUST。
- 移行で得たドキュメント群から、コレクション（3.5）の初期版を組み立てる SHOULD。

> 注: 移行は語彙に綺麗に収まらない雑多な入力を相手にするため、POC の素直なサンプルより
> 遵守・マッピングが難しくなる可能性が高い。実データでの遵守度はまだ未検証であり、次のPOC候補。

## 9. オープンソース化に向けて（informative）

Ordito を規格として公開するための構成案。GitHub 上に一通りのワークフローを揃えて置く。

- **規格本体**: 3章（IR・コレクション）と4章（契約）と7章（スキル契約）が規格のコア。
- **参照実装**: 生成エンジン・デフォルトテンプレート・代表的なスキルを別ディレクトリ／別リポジトリに置く。
  POC実装（Node.js 依存ゼロ）がその出発点となる。
- **準拠テスト**: IR / 契約 / コレクションの **JSON Schema（draft 2020-12）バリデータ**（3.3・3.5）と、
  サンプル IR → 期待される成果物の検証セット、および検証の二層（6章）を提供する SHOULD。
  ブロック語彙の JSON Schema は POC で書き起こし済みで、これを準拠テストの中核に据える。
- **バージョニング**: 規格はセマンティックバージョニングに従う。IR / 契約スキーマの破壊的変更は
  メジャーバージョンを上げる。
- **当面の目標**: POC を二周完了（単一ページ / 複数ページ＋コレクション＋混在生成）。仕様の骨格と
  主要な穴（フィールド欠落・サイト構造・枠内リンク・パス規則）は埋まった。次は実データ移行での
  遵守度検証（手元に実データが揃い次第）か、差分更新・二段確認 UX の実装へ。

  見出しのみ項目。
- **`field_map` の構造化**（4.4）: 現状フリーテキスト＋`OMIT` 予約語。`{ target, omit? }` への構造化の是非。
- **未マップ検出と語彙の単一ソース化**（4.4・3.3）: 検出対象フィールドをブロック JSON Schema から引く統合。
- **ナビ未掲載ページの限定生成**（3.7）: 「生成はするがナビに出さない」明示フラグの設計。
- **実データ移行での契約遵守度の検証**（8章）: 素直なサンプルではなく雑多な実データでの遵守度（実データ待ち）。
- **意味チェックの標準的手段**（6.2）: LLMジャッジ以外の選択肢（決定論的な差分照合の範囲）。
- **差分更新・二段確認 UX の実装**（3.4・5.4・7.3）: 設計は固まったが未実装。
- **公開時のライセンス選定**。

## 付録A: 用語

- **Ordito**: 本システム／規格の名称。織物の経糸に由来。
- **IR（中間表現）**: 表示から独立した本文の構造化データ。正本は JSON。
- **コレクション（サイトマップ）**: ナビ項目・順序・グループなど、サイト全体の構造を表す上位概念（3.5）。
- **枠 / テンプレート**: レイアウト・デザイントークン・コンポーネントを固定で提供するもの。
- **テンプレート契約**: AIと検証器に渡す、使用可能なコンポーネント・許可リスト・フィールドマッピングの定義（4.2）。
- **語彙**: 使用可能なブロック型とコンポーネントの集合。
- **render_hint**: 期待 HTML 構造の例。詳細度で許容する揺らぎを制御する（4.3）。
- **field_map**: ブロック型の各フィールドの行き先の網羅定義。データ欠落を防ぐ（4.4）。
- **スキル / プラグイン**: ストアに対する書き込みまたは読み出しの拡張。
- **制約された多様性**: 固定された枠の内側で、AIの出力に揺らぎを許す設計方針。情報欠落は含まない。
- **データ更新と生成の分離**: IR の書き換えと成果物の生成を別工程とし、生成を明示トリガーで行う方針。
- **孤立ページ**: コレクションとIRのずれで生じる、参照先欠落（目次にあるがページ無し）または
  ナビ未掲載（ページはあるがナビに無し）の状態（3.7）。
- **枠内リンク**: テンプレートの枠が持つホーム・ロゴ等のリンク。実体パスは生成時に現在ページ基準で相対解決する（4.7）。
