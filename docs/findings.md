# Ordito POC 所見メモ（仕様 v0.3 への材料）

POC 実装（一周: IR → 生成 → 枠に収まった HTML）を通して判明した、仕様の不明点・矛盾・
決め打ち箇所をまとめる。特に **テンプレート契約のフォーマットと AI への渡し方** に重点を置く。

- 対象仕様: v0.2（当時）。現行は [../spec/ordito-spec.md](../spec/ordito-spec.md)、旧版は [../spec/history/](../spec/history/)
- 実装スタック: Node.js v18+（依存ゼロ）
- 検証日: 2026-06-13

## 1. スタック選定とその理由

**Node.js v18+（依存ゼロ）** を選択。

- JSON が第一級（`JSON.parse`/`stringify`）で、仕様の「JSON 統一」原則に摩擦なく従える。
- テンプレートリテラルで HTML 文字列の組み立てが容易。
- `fetch` 標準搭載で、Anthropic API 呼び出しに外部依存（`npm install`）が不要。
- ビルド工程ゼロで `node` だけで動くため、POC の「最短で一周」に最適。

> 代替候補だった Python も JSON は得意だが、HTML 文字列組み立てと AI 呼び出しの取り回しで
> Node のテンプレートリテラル＋標準 `fetch` がわずかに有利と判断した。

## 2. 仕様の不明点・矛盾と、置いた仮定

### 2.1 ブロックのフィールド詳細が未確定（§3.3 / §10 TBD）

§3.3 は「語彙（type の集合）は確定、内部スキーマは詰める」とある。POC では次のように**決め打ち**した。
これは v0.3 でスキーマ化すべき第一候補。

| type | 決め打ちした形 |
|------|----------------|
| `heading` | `{ level: 1-6, text }`（描画時 h1-h4 にクランプ） |
| `paragraph` | `{ text }`（インライン記法 `` `code` `` / `[label](href)` / `**bold**` を許容） |
| `code` | `{ lang, text, filename? }` |
| `inline_code` | `{ text }`（単独ブロックとしても許容。主用途は段落内インライン） |
| `params` | `{ items: [{ name, type, required, desc, default? }] }` |
| `table` | `{ headers: [...], rows: [[...]] }` |
| `note` | `{ variant: info\|warning\|danger, text }` |
| `list` | `{ ordered: bool, items: [string] }` |
| `link` | `{ href, text }` |
| `tabs` | `{ tabs: [{ label, blocks: [...] }] }`（**ブロックの入れ子**を許容＝再帰構造） |
| `steps` | `{ items: [{ title?, text } \| string] }` |

**論点（v0.3向け）:**

- **インライン記法の規格化**: `paragraph.text` に「インライン記法を許容」（§3.3）とあるが、記法自体が未定義。
  POC は Markdown 風（`` `code` `` / `[label](href)` / `**bold**`）を採用した。IR は「HTML を含まない MUST」（§3.1）
  なので、インラインも HTML ではなく**規格化された軽量マークアップ**で持つ必要がある。これは規格本体で定めるべき。
- **`tabs` の入れ子**: 「言語別例など」とあるためタブ内に `code` 等を入れたい。POC では `tabs[].blocks[]` で
  ブロックを再帰的に持たせた。ブロックがネスト可能かどうかは規格で明示が必要（差分更新の単位 block.id とも関係）。
- **`inline_code` をブロック型に含める意味**: §3.3 の表にあるが「段落内」が主用途。独立ブロックとしての
  用途が薄い。語彙から外し paragraph のインライン記法に統合する案を v0.3 で検討。
- **`link` の href**: 内部ドキュメント参照（doc id）か外部 URL かの区別が要る。POC は「既知の doc id なら
  `<slug>.html` に解決、それ以外は素通し」とした。IR は表示非依存なので、href は **doc id（論理参照）**で
  持ち、HTML パスへの解決は生成時に行う、という分離が妥当。規格で「内部参照は id、外部は URL」と定めるべき。

### 2.2 「単一ページ」と「ナビ」の所有権が曖昧（§4.1 / §4.3）

§4.1 は「ナビ・サイドバーはテンプレートが固定で持つ MUST」とする。一方で、ナビ項目（どのページがあるか）は
**ドキュメント集合**に依存し、単一 IR だけでは決まらない。POC では次のように**決め打ち**:

- ナビの**構造とスタイル**はテンプレート（`frame.html` / `styles.css`）が所有。
- ナビの**項目**は生成エンジンがドキュメント集合（`samples/*.json`）から組み立て、現在ページに `aria-current` を付与。

→ v0.3 では「テンプレートが所有する枠」と「サイト全体（ドキュメント集合・ナビ・順序）」を別レイヤとして
定義すべき。IR の `meta.order` はナビ順に使えるが、サイト構造を表す上位の概念（サイトマップ／コレクション）が
仕様に欠けている。

### 2.3 検証（§6）は「省略可」だが契約遵守の計測に必須

指示で検証は POC 省略可。だが本 POC の主目的「契約を守った HTML が返るか」を**測る**には、最小の遵守チェックが
事実上必要だったため、`engine/validate.js` を実装した（許可クラス／タグ・style/id 禁止・枠侵食・整形式の近似）。
→ 検証は「公開前ゲート」であると同時に「AI 生成の品質計測器」でもある。v0.3 で §6 にこの二役を明記すると良い。

### 2.4 `content_slot` の解釈（§4.2 / §5.1）

`content_slot: "#doc-body"` は「本文を差し込む領域」だが、AI 出力が **slot の内側 HTML だけ**なのか、
slot 要素自体を含むのかが曖昧。POC は「**内側のフラグメントのみ**」と解釈（枠＝テンプレート所有の原則と整合）。
v0.3 で「AI 出力は content_slot の innerHTML である」と明記すべき。

## 3. テンプレート契約のフォーマットについての所見 ★重点

§4.2 の最小例（`template_id` / `components` / `content_slot` / `constraints`）は出発点として妥当だが、
**実装が契約を機械的に使う**には情報が足りなかった。POC で次を拡張した（すべて v0.3 への提案）。

| 追加フィールド | 目的 | なぜ必要か |
|----------------|------|-----------|
| `allowed_classes`（フラットな許可リスト） | 検証器がクラスを照合するため | components を走査して導出もできるが、検証の単一ソースとして明示が安全 |
| `allowed_html_tags` | 許可タグの照合 | 「枠侵食禁止」を機械判定するには許可タグ集合が要る |
| `allowed_attributes` | 属性ホワイトリスト | `style`/`id` 禁止を肯定形（許可リスト）で表現できる |
| `components[].render_hint` | 期待 HTML 構造をAIに具体例で示す | クラス名だけでは「どの要素に・どう入れ子で」付けるか曖昧 |
| `components[].variants` / `variant_class` | `note` 等の variant をクラスに展開する規則 | `doc-note--{variant}` の生成規則が無いと AI が独自命名する |
| `components[].sub_classes` | 子要素のクラス（例 `doc-params__name`） | コンポーネント内部構造のクラスも許可リストに必要 |
| `constraints` の肯定/否定の明示化 | `must_render_all_blocks` 等の機械可読フラグ | §4.2 例は禁止2項目のみ。遵守チェックに使える粒度に拡張 |

**所見（v0.3向け）:**

- §4.2 の `components[].class` だけでは、**HTML の入れ子構造**（`<div class="doc-code"><pre><code>`）を
  AI が一意に再現できない。`render_hint`（期待構造のテンプレート文字列）が遵守度を大きく左右した（§4 参照）。
- 契約は「**許可リスト（ホワイトリスト）**」として表現するのが、検証とも AI 指示とも相性が良い。
  禁止リスト（MUST NOT の列挙）だけだと、検証器が「何が許されるか」を別途持つ必要があり二重管理になる。
- `variant` のクラス展開規則（`variant_class` テンプレート）は明示が必須。無いと AI が `doc-note-warn` 等の
  独自命名をしがち。

## 4. AI への契約の渡し方 — 4戦略の比較実験 ★最重点

POC で最も検証したかった点。「テンプレート契約をどういう形でモデルに渡すと、契約を守った HTML が
返ってくるか」を実測した。

### 4.1 実験方法

- 同一の `contract.json` ＋ IR(JSON) に対し、**プロンプトの枠組み（戦略）だけ**を変えた4種を比較。
  プロンプトはエンジン自身の `engine/prompt.js` が生成（＝本物のエンジンプロンプトを検証）。
  - `rules`: MUST/MUST NOT を番号付きで明示列挙 → 契約JSON → IR。
  - `schema`: 「使ってよいコンポーネント表」を主役にした許可リスト型（プロンプト最小）。
  - `example`: 1ブロックの正解HTMLを見せてから残りを依頼。
  - `minimal`: 足場なし。契約JSON＋IR＋一文の指示のみ（ベースライン）。
- 生成モデルは production 相当の **Claude Sonnet** で固定（戦略以外を統制）。2ドキュメント × 4戦略 = 8生成。
- 二段で遵守度を測定: **(A) 決定論バリデータ**（`engine/validate.js`・客観: 許可クラス/タグ・style/id・枠侵食・整形式）
  ＋ **(B) LLMジャッジ**（adversarial・IR忠実度/順序/捏造を検査）。

### 4.2 結果

| 戦略 | 平均プロンプト長 | LLMジャッジ平均(0-100) | 客観バリデータ | 全ブロック描画/順序保持 |
|------|------------------|------------------------|----------------|--------------------------|
| `schema` | **約 5.8KB** | 99 | ✅ 8/8 合格 | ✅ / ✅ |
| `example` | 約 10.0KB | 99 | ✅ 8/8 合格 | ✅ / ✅ |
| `minimal` | 約 9.4KB | 99 | ✅ 8/8 合格 | ✅ / ✅ |
| `rules` | 約 10.9KB | 98 | ✅ 8/8 合格 | ✅ / ✅ |

- **8生成すべてが客観バリデータに合格**（許可外クラス・許可外タグ・style/id・枠侵食ゼロ）。
- LLMジャッジが挙げた「指摘」はすべて**非違反の確認記録**だった（実害なし）。

### 4.3 わかったこと（結論）

1. **契約 JSON をそのまま渡す方式は機能する。** 仕様 §4.2 の「契約は JSON で渡す」は、能力の高いモデルなら
   足場が薄くても（`minimal`でも）契約を守れた。**契約が JSON で機械可読であること自体が効いている。**
2. **`schema` 戦略（許可リスト=表を主役）がコスト効率で最良。** プロンプト長が `rules` の約半分で、遵守度は同等以上。
   → リファレンス実装のレベル2プロンプトは **許可リスト型を既定**にするのが妥当（`engine/prompt.js` の `schema`）。
3. **遵守度を最も左右したのは `render_hint`（期待HTML構造の例示）。** クラス名の列挙だけでは
   `<div class="doc-code"><pre><code>` のような**入れ子構造**を一意に再現できない。構造を文字列例で示すと安定した。
   → §4.2 の `components` に `render_hint`（または同等の構造例）を**規格として加える**ことを推奨。
4. **検証の二層が有効。** 客観バリデータ（決定論・安価）で機械的違反を弾き、LLMジャッジで意味的忠実度を見る、
   の二段構成が POC でうまく働いた。§6 の検証は「機械チェック＋意味チェック」の二層で規定すると良い。

### 4.4 契約設計の落とし穴 — `default` の沈黙脱落 ★重要

ジャッジが全 `api-auth` で一貫して検出した、実害ある所見:

- IR の `params` items には `default`（`scope="read"`, `expires_in="3600"`）があるが、**AI 出力では脱落した**。
- 原因: 契約の `params` コンポーネントは「名前/型/必須/説明」の4列しか定義しておらず、`default` の扱いが**契約に無い**。
  AI は `render_hint` に忠実に従った結果、IR に存在する情報を落とした。
- 一方、リファレンス実装の決定論レンダラ（`render.js`）は `default` を説明列に追記する独自実装になっており、**両者が分岐**した
  （スクリーンショット比較で `expires_in=3600` が決定論版にだけ出ている）。

**教訓（v0.3向け）:** 契約のコンポーネント定義が IR の全フィールドの行き先を明示しないと、**IR の情報が静かに失われる**。
これは「制約された多様性」の許容範囲ではなく**データ欠落**。規格として「契約はブロック型の各フィールドの
マッピングを網羅的に定義する MUST／未マップのフィールドは検証が警告する SHOULD」を加えるべき。

なお POC では、決定論レンダラ（`default` を説明列に追記）と契約 `render_hint`（`default` に言及なし）が
意図せず分岐している。これはまさに「契約がフィールドの行き先を定義していない」ことの帰結であり、
本メモの所見を生んだ実物の証拠としてそのまま残す。どちらに揃えるか（＝`default` をどう表現するか）は
v0.3 で契約スキーマを確定する際に決める設計課題。

### 4.5 さらなる構造的差異 — AI と決定論レンダラ（いずれも契約遵守）

`params` テーブルを AI 版（schema戦略）と決定論版で比較すると、**契約のクラスは双方守りつつ、内側の
HTML構造が分岐**した。

| セル | 決定論版 | AI版（schema） |
|------|----------|----------------|
| 型 | `<td><code class="doc-ic">string</code></td>` | `<td>string</td>`（code 包みなし） |
| 必須 | `<td><span class="doc-params__req">必須</span></td>` | `<td class="doc-params__req">必須</td>`（span ではなく td にクラス） |

- 原因: 契約の `params` の `render_hint` は名前セルしか構造を示さず、型・必須セルの内側構造を `...` で
  省略していた。**未指定の部分で AI とリファレンス実装が別々の妥当な選択をした**結果。
- どちらも `validate.js` を通過し、ブラウザ描画も崩れない（許可クラス・許可タグの範囲内）。
  これはまさに仕様の「制約された多様性」が機能している実例。
- 一方で、ページ間/レンダラ間の構造一貫性を上げたいなら **`render_hint` を全セル分まで具体化**すれば
  揺らぎを縮められる。**`render_hint` の詳細度が、許容する揺らぎの量を直接コントロールする**——
  これは §4.3 の結論3を補強する重要な観察。v0.3 では「契約の `render_hint` は各ブロック型の
  全フィールドの行き先を構造込みで例示する SHOULD（詳細度で揺らぎを制御する）」と明記すると良い。

## 5. その他の決め打ち一覧

- 出力 HTML は **CSS をインライン化した自己完結の単一ファイル**にした（`file://` で確実に開けるため）。
  本番では外部 CSS 参照（キャッシュ効率）に切り替えるべき。
- 出力ファイル名は doc id の**末尾セグメント**（`guides/quickstart` → `quickstart.html`）。
  id 全体をパスにする方式（ディレクトリ階層維持）は POC では未対応。
- `meta.generated_at` は出力に情報として刻むのみ。`updated_at` との比較による再生成判定（§3.4 / §5.4）は対象外。
- 未知ブロック型は `note(warning)` にフォールバック描画（§3.3 の SHOULD に準拠）。
- レベル2（AI）では**本文領域全体**を AI に生成させた。§5.2 推奨の「構造化ブロックは機械的、本文は AI」の
  混在は POC では未実装（比較しやすさを優先）。本番では params/table/steps を決定論、prose を AI が妥当。

## 6. 多エージェント・レビューで発見・修正した実装バグ

POC 実装後、独立したレビュー（4観点×検出 → adversarial 再検証）を回し、確定した指摘を修正した。
記録として残す（リファレンス実装の品質と、検証工程の有効性の裏付け）。

| 重大度 | 箇所 | 内容 | 対応 |
|--------|------|------|------|
| major | `validate.js` | `style=`/`id=` をエスケープ済みテキスト（HTMLを説明するコードブロック）でも誤検出 → 決定論レンダラが自前検証で落ちる | **修正**: 属性検査を「実開始タグの内側」に限定。コード本文中の `style=` を誤検出しなくなった |
| major | `render.js` | `inline()` が href を二重エスケープ／エスケープ後の href を解決に渡していた | **修正**: コードスパンとリンクを先に退避し、href は raw で解決・属性は一度だけエスケープ |
| major | `render.js` | インラインリンクのパーサがコードスパン内の `[..]` を誤変換 | **修正**: コードスパンをプレースホルダ退避してから link/bold を適用 |
| major | `prompt.js` | `example` 戦略が型不一致・未エスケープの誤った例を提示していた | **修正**: 実際の `renderBlock` で例を生成（契約準拠・エスケープ済み） |
| major | `generate.js` | `meta` 欠落の IR で実行全体が TypeError で停止 | **修正**: `meta`/`blocks` を読み込み時に正規化、`id` 必須チェック |
| major | `provider.js` | `stripFences` が本文中のフェンスも剥がし HTML を切り落とす恐れ | **修正**: 「全体が単一フェンス」のときのみ剥がす。`html` 欠落時は明示エラー |
| minor | `generate.js` | slug 衝突（異なる id が同じ出力名）で黙って上書き | **修正**: 衝突を検出してエラー終了 |
| minor | `generate.js` | フラグの値欠落・不正な mode/strategy が生スタックトレースで落ちる | **修正**: 引数検証を追加し明示エラー |
| minor | `validate.js` | 許可属性リストを強制せず `target`/`onclick` 等が素通り | **修正**: `allowed_attributes` を許可リストとして強制 |
| minor | `generate.js` | `report.json` のパス表示が `--out` を無視・決定論時も strategy を出力 | **修正**: 実際の出力先を表示、決定論時は strategy を null |
| minor | `validate.js` | タグ名正規表現がハイフン付きタグを途中で切る | **修正**: タグ名に `-` を許可 |
