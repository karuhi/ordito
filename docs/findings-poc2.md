## 1. 決め打ちした4つの TBD（実装で出した答え）★最重要

### (a) ブロックの JSON Schema 化（§3.3 / §10）

**実装**: `schemas/ordito-v0.3.schema.json`（draft 2020-12）に全ブロック型を機械可読化。検証器は依存ゼロの
最小バリデータ `engine/schema-check.js`。生成エンジンの機械チェック層にも統合（IR を生成前に検証）。

決め打ちした点と理由:

- **判別方式**: 各ブロック型を `$defs/blocks/<type>` に定義し、`type` を `const` で固定、`additionalProperties:false`。
  `$defs/block` を全型の `oneOf` とした。`const`＋`additionalProperties:false` により oneOf が常に1件だけ一致＝
  実質的なディスクリミネータになる（OpenAPI の discriminator 構文は使わず標準語彙で表現）。
- **`default` の型**: `params.items[].default` は `["string","null"]`（値か、明示的な null）。null を「既定なし」として
  許容するのは §3.2 の例（`"default": null`）に合わせた。
- **`steps.items` の略記**: `oneOf: [string, {title?, text}]`（文字列だけの手順も許容）。
- **再帰**: `tabs.tabs[].blocks[]` は `$ref: #/$defs/block` で再帰（§3.3.3）。
- **最小バリデータの実装範囲**: type(配列可)/required/properties/additionalProperties:false/items/enum/const/
  minimum/maximum/oneOf/ローカル$ref のみ。pattern・format・$id跨ぎ参照などは未実装（POCの割り切り）。
  oneOf で0件一致のときは「最も近い分岐（エラー最少）」の詳細を出して原因を可視化した。
- 提案: 規格本体は `(a)` の JSON Schema を**準拠テスト**（§9）の一部として同梱すると良い。

### (b) コレクションの詳細スキーマ（§3.5 / §10）

**実装**: `$defs/collection` ＋ `$defs/navItem`。`nav[]` の各項目は2種の `oneOf`:

- ドキュメント参照: `{ doc, order?, label? }`（`label` はナビ表示名の上書き。省略時 `meta.title`）
- グループ: `{ group, order?, items[] }` — `items` は `navItem` の**再帰**＝**多階層グループ**を表現。

決め打ちした点と理由:

- **並び順**: 各階層内で `order` 昇順、`order` 省略時は配列の出現順（安定ソート）。`order` を任意にしたのは、
  手書きで楽に並べられるようにするため。IR の `meta.order` は使わず**コレクションの order を最終決定**とした（§3.5の所有分離）。
- **多階層**: グループの `items` に再びグループを置けるようにした（サンプルで「API リファレンス > 詳細 > ページネーション」を実証）。
- **所有の分離（実証）**: ナビの**項目・順序・グループ**＝コレクション（`collection.js` が解釈）、**描画構造・スタイル**＝
  テンプレート（CSS の `site-nav__group` 等）。エンジンが現在ページの `aria-current` とリンクの相対化を合成。
- 未対応（v0.4候補）: 複数コレクション間の関係、外部リンクをナビ項目に置くこと、ナビ項目への doc 以外（見出しのみ等）。

### (c) field_map の表現と「表示しない」の明示（§4.4 / §10）

**実装**: 契約の各 `components[].field_map` が、そのブロック型の**意味フィールド**→行き先を網羅定義。

- **暗黙の脱落を禁じる方法**: 表示しないと決めたフィールドは値 `"OMIT"` を明示する（契約の `field_map_conventions` に規定）。
  検証器（`validate.js` の `validateFieldMap`）は、IR ブロックに存在する意味フィールドのうち `field_map` に**キーが無い**ものを
  `unmapped_field` 警告にする。`"OMIT"` はキーが在る＝マップ済み扱いなので警告しない。
- **実証**: `params` の `field_map` から `default` を外すと、`params.default が field_map に未定義` を即警告（＝第1弾で
  沈黙脱落した `expires_in=3600` のケースを v0.3 のガードが捕捉）。完備時は警告ゼロ。出力でも `default` 列が保持される。
- 決め打ち: 「意味フィールド」の集合はブロック型ごとに固定リストで持った（`validate.js` の `mappableFields`）。
  そのため**リストにない全く新規のフィールド**は検出対象外（既知だが未マップなフィールドのみ検出）。
  完全な検出にはブロックの JSON Schema からフィールドを引くのが理想（(a) と統合する余地・v0.4）。
- 提案: `field_map` の値は現状フリーテキスト（行き先の説明）。`"OMIT"` だけ予約語にした。v0.4 では
  `{ target: "<class/attr>", omit?: true }` の構造化も検討に値する。

### (d) 出力の物理パス規則と内部リンク解決（§3.3.2 / §10）

**実装**: `engine/paths.js`。doc id の**階層を保持**して物理パスにする。

- `guides/quickstart` → `site/guides/quickstart.html`、`reference/pagination` → `site/reference/pagination.html`。
- **内部リンク解決**: link.href / インラインリンクの内部参照（doc id）を、**現在ページからの相対パス**へ解決
  （`relHref(fromId, toId)`）。深さが違っても `file://` で開ける（例: `guides/` から `reference/` へは `../reference/...`）。
- 外部参照（スキーム付きURL・`//`・`#`）は素通し。未知の内部参照は `unresolved_ref` 警告。
- **テンプレート枠内のリンクも相対化が必要だった（新発見）**: ヘッダーのブランドリンク `index.html` が、
  サブディレクトリのページから解決できずリンク切れになった。`{{HOME}}` プレースホルダを足し、エンジンが
  `relHref(currentId, "index")` で各ページ向けに埋める形に修正。→ **「枠が固定」でも、枠内のリンクはページ深さに依存する**。
- 末尾セグメントのみ（フラット）方式を採らなかった理由: doc id が `guides/quickstart` と `reference/quickstart` のように
  別階層で同名末尾になると **slug 衝突**する。階層保持なら衝突しない。代償（相対パス計算）はエンジンが吸収する。
- 第1弾互換: `--collection` 無しのフラットモードは従来どおり末尾セグメント＋ルート直下出力を維持（後方互換）。

## 2. 複数ページ・ナビ・リンク・混在生成で判明した新たな所見（v0.4向け）

- **枠内リンクの深さ依存（上記 d）**: §4.1「ナビ等は枠が固定」は描画**構造**の話だが、枠が持つ**リンク**（ホーム・ロゴ等）は
  出力パスの深さに依存する。規格に「テンプレート枠内のリンクも、生成エンジンが現在ページ基準で相対解決する MUST」を加えるべき。
- **混在生成の粒度（§5.2）**: レベルは**トップレベルのブロック単位**で割り当てた（paragraph/note=L2、他=L1）。
  `tabs` のような入れ子ブロックは、コンテナのレベルに従って描画（=決定論）とした。仕様は「ブロックごと」と言うが、
  **入れ子ブロックのレベルを個別に選べるか**は未定義。v0.4 で「入れ子はコンテナのレベルに従う／個別指定可」を明記すべき。
- **L2 のAIへの渡し方（混在時）**: 散文ブロックだけを渡し、`{id: html}` のマップで返させ、エンジンが本文順に差し込む方式。
  リンクは **doc id のまま返させ、解決はエンジン**（プロンプトで明示が必要）。これにより「IR=論理参照／生成=物理解決」の
  分離が AI 経路でも保てた。所見: AI に「href を変換するな」と明示しないと、気を利かせて `.html` を付けることがある。
- **意味チェック層の費用対効果（§6.2）**: 構造化を決定論にした分、AI が触る面積が小さく、意味チェック（忠実度）の対象も
  散文に絞れる。混在は「揺らぎを許す箇所を最小化しつつAIの価値（自然な散文）を活かす」現実的な落とし所だと確認できた。
- **コレクションとIRの整合**: コレクションが参照する doc id に対応する IR が無い場合・IR があるがコレクションに無い場合の
  扱いは実装依存（前者は警告してスキップ、後者は生成対象外とした）。v0.4 で「孤立ページ」の扱いを規定すると良い。

## 3. レビューで発見・修正した実装バグ（adversarialレビュー結果）

第2弾実装後、多エージェント・レビュー（仕様遵守・コード・IR忠実度×5ページ・TBD妥当性 → adversarial検証、
計31エージェント）を回した。確定 **major 3 / minor 8 / nit 10**。主要なものと対応:

| 重大度 | 箇所 | 内容 | 対応 |
|--------|------|------|------|
| major | `render.js` | `inline()` でリンクラベル内のコードスパンが NUL プレースホルダのまま出力に漏れる | **修正**: コード復元をリンク復元の後に回し、label 内の退避分も一括復元 |
| major | `generate.js` | `resolveFragmentHrefs` がエスケープ済み href（外部URLの `&amp;` 等）を二重エスケープ | **修正**: 解決結果が原値と同じなら原文のまま返す |
| major | `schemas/...json` | `paramItem.default` が string\|null 限定で §4.4 看板例 `expires_in=3600`（整数）を表現不可 | **修正**: default を string/number/boolean/null に拡張。サンプルの既定値を数値に戻した |
| minor | `validate.js` | §4.4 未マップ検出が固定リスト駆動で、**未知フィールド**の脱落を取りこぼす | **修正**: ブロックの実キーから動的導出（params/steps/tabs は item キーの和集合）。未知 `deprecated` 等も検出 |
| minor | `generate.js` | §6.2 意味チェック（IR忠実度）層がエンジン未実装（findings の「✅」と不一致だった） | **修正**: 最小・決定論版 `validateFidelity`（IR素テキストの出力反映を照合）を追加。LLMジャッジ層はレビューWorkflowで別途実施 |
| minor | `collection.js` | nav に同一 doc が複数回 → ページ重複生成 | **修正**: `[...new Set(collectDocIds())]` で一意化 |
| minor | `collection.js` | doc でも group でもない不正 nav 項目を空グループ描画 | **修正**: group のときのみ描画、不正項目はスキップ |
| minor | `generate.js` | doc id `"index"` がホーム `index.html` と衝突 | **修正**: 予約 id として拒否 |
| minor | `validate.js` | `"OMIT"` センチネルのコメントが実装（値は不問）と齟齬 | **修正**: コメントを実態（field_map にキーが在ればマップ済み／OMIT は人間向け慣習）へ |
| nit | `generate.js` | `runFlat` が警告を標準出力に出さない（`runCollection` と非対称） | **修正**: 警告表示を両経路で統一 |
| nit | `schemas/...json` | `$defs/block` の `required` が各分岐と二重で診断重複 | **修正**: block 直下の required を除去 |

**コード修正せず記録に留めたもの（理由あり）**:

- (minor) 仕様 §10 のパス規則 TBD 行が「POCは末尾セグメント」のまま未更新で、本実装の階層保持と矛盾
  → **これは規格本体（`ordito-spec-v0.3.md`）の記述であり、勝手に書き換えない**。v0.4 改訂時に §3.3.2 付近へ
  「物理パスは doc id の階層を保持。フラット方式は slug 衝突のため不採用」と確定記述へ移すことを提案（上記 §1(d)）。
- (nit) 第1弾サンプル `samples/api-auth.json` が v0.3 削除済みの `inline_code` を使用 → v0.3 スキーマは**正しく不適合と判定**
  （望ましい挙動）。第1弾の歴史的資産として残置。`render.js` は後方互換で描画。
- (nit) `heading.level` はスキーマで 1–6、描画時に h1–h4 へクランプ → これは仕様 §3.3 の記述どおりで矛盾なし。
- (nit) ナビの「生成」時刻はビルド時刻で、IR の `generated_at`(null) とは別物（本文領域外のクローム）。IR 忠実度には影響しない。

> IR忠実度（§6.2 意味チェック）はレビューWorkflowの5ページ判定で**全ページ合格**（全ブロック・全フィールド反映、
> 順序保持、捏造なし）を確認。加えてエンジンに最小の決定論 fidelity チェックを常設し、決定論・混在とも警告ゼロ。
