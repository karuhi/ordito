# Ordito スキル化 所見メモ（仕様 v0.5 への材料）

差分更新（§3.4）・二段確認（§7.3）を `.claude/skills/` の原子スキル群として実装して判明した、
仕様の不明点・新たな穴・決め打ち。これまでの findings と同じ粒度。

- 対象仕様: [../spec/ordito-spec.md](../spec/ordito-spec.md)（v0.4）
- スタック: Node.js 依存ゼロ（継続）。スキルはリファレンス実装エンジンを呼ぶ。
- 検証日: 2026-06-14

---

## 0. 実証できたこと

[skills-two-stage-demo.md](skills-two-stage-demo.md) のトランスクリプトのとおり、
**原子スキルの組み合わせで AI エージェントが二段確認を組み立てられる**ことを実証した。
更新→記載確認→未反映検出→反映確認→該当ページのみ再生成→検証、が通る。

---

## 1. 「確認の主体を AI に委ねる」設計は §7.3 と整合するか

**整合する。むしろ §7.3 の含意を明確化する。** §7.3 は「システムが問う:『記載しますか？』」と書くが、
**「システム」が誰か**は曖昧だった。本実装の答え:

- 確認（y/n の問い）は **AI エージェント**が行う。スキルは**非対話・冪等・JSON 入出力**に徹し、確認の判断を持たない。
- 「記載しますか？」＝ `ordito-update-block` を呼ぶ前の確認、「反映しますか？」＝ `ordito-generate` を呼ぶ前の確認。
  どちらも agent がユーザーに尋ね、yes のときだけ対応スキルを呼ぶ。
- **更新スキルと生成スキルを分けたこと自体が §5.4 の実装**。`ordito-update-block` は常に `generated:false` を返し、
  生成は `ordito-generate`（明示トリガー）でしか起きない。二段確認の「二段」は、この2スキルの境界に対応する。

**v0.5 提案**: §7.3 に「確認はオーケストレータ（AIエージェント等）の責務であり、スキルは非対話とする SHOULD。
スキルは状態を返し、確認の判断は呼び出し側が持つ」と明記する。これで「システム」の曖昧さが解消する。

## 2. スキルの粒度は適切か

4つ（update / detect-stale / generate / validate）は **書き込み1・読み出し2・検証1** に対応し、過不足は少ない。
気づいた論点:

- **`ordito-generate` が2つの責務を持つ**: 「エンジン起動」＋「`generated_at` 更新」。だが後者は「生成が完了した」
  事実の記録であり、分離すると `generated_at` の更新漏れ（生成したのに未反映のまま）を招く。**一体が正しい**と判断。
- **`ordito-update-block` に dry-run が無い**: 現状は「確認 → 書き込み」で、書き込みが確定操作。agent が
  「こう変わります」を**書き込み前に**見せたい場合、レンダリング差分のプレビューが欲しい。
  → v0.5 候補: `update-block` に `"dry_run": true`（before/after だけ返し書き込まない）を追加。
  現状は `changed:false` の冪等性と before/after 返却で代替できるが、プレビュー専用フラグがあると二段確認の一段目が滑らか。
- **粒度の最小性**: 「未反映の有無」と「未反映の一覧」を分ける必要は無く、`detect-stale` 1つで両用途を満たせた。

## 3. スキル間 JSON 契約の過不足

- **十分だった点**: `detect-stale` が timestamps を返す → 確認文面（「updated > generated」）に使える。
  `update-block` が before/after・changed を返す → agent が差分提示・冪等判定に使える。
  `generate` が generated[] と検証 warnings 件数を返す → 反映結果の要約に使える。
- **不足／曖昧だった点（v0.5 候補）**:
  - **スキル I/O 自体の JSON Schema が無い**: IR・契約・コレクションはスキーマ化したが、スキルの入出力契約は
    SKILL.md の散文止まり。準拠スキルを他者が作るなら、スキル I/O も JSON Schema 化すべき（§7 のスキル契約の延長）。
  - **`generated_at` の真実が二重**: IR の `meta.generated_at`（detect-stale が使う論理状態）と、出力 HTML の
    フレームに表示される「生成時刻」（ビルド時刻のクローム）が別物。混同しやすい。出力の表示は IR の
    `generated_at` を反映するか、明確に「ビルド時刻」とラベルするか、v0.5 で統一すべき。
  - **検出と生成の間の競合**: `detect-stale` → （ユーザー確認の間に IR が変わる）→ `generate` の間に
    TOCTOU がありうる。POC では無視（単一ユーザー前提）。マルチエージェント運用なら版番号/ロックが要る（TBD）。

## 4. 決め打ち

- **タイムスタンプはミリ秒精度**: 秒精度だと連続 update→generate が同秒になり `updated_at == generated_at` で
  未反映判定が崩れた（実測）。`store.nowIso()` はミリ秒を保持する。仕様は精度を規定していない →
  v0.5 で「比較に足る精度（ミリ秒以上）を持つ ISO8601」を SHOULD 化すべき。
- **未反映判定は厳密 `>`**: `generated_at < updated_at` のみ未反映。等値は「同時刻に生成済み」とみなし未反映にしない。
- **`generated_at` はクロックスキュー耐性のため `max(now, updated_at)` で押印**（レビューで発覚）: `updated_at` は
  任意のソース（別マシン・未来日時のデータ）由来になりうる。`generated_at = now()` だと、`now < updated_at` のとき
  生成しても未反映が永久に解消しない。`max(now, updated_at)` なら「その `updated_at` の内容を反映済み」を必ず満たす。
  → **v0.5 提案**: §3.2/§5.4 で「生成器は `generated_at` を、反映した IR の `updated_at` 以上の値で記録する MUST
  （`max(生成時刻, updated_at)`）。素朴な現在時刻だとクロックスキューで未反映が解消しない」と明記すべき。
  これは「`updated_at` が信頼できる単調時計を前提にしている」という、仕様の暗黙の前提を露わにした。
- **`patch` はトップレベル浅マージ**（配列は丸ごと置換）。ブロック単位更新（§3.4）の粒度として妥当。ネスト内の
  1要素だけの更新は、対象ブロック（例 params）を丸ごと patch する運用。
- **IR ストアの場所は `ir_dir`（既定 `samples/ir`）**。ファイル位置は doc id と独立（id は論理パス）で、id で引く。
- **スキルはリファレンス実装エンジンを `spawn` で呼ぶ**（`ordito-generate`）。エンジンを黒箱として扱い、スキル境界を保つ。

## 5. スコープ外（指示どおり）

- 知見の自動抽出（「溜まりました」の検知）は対象外。更新内容はスキルへの入力として与える前提。
- Web/リッチUI 無し。確認は agent の対話に委ねる。
