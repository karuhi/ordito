# Contributing to Ordito

Ordito は「規格（spec）」と「参照実装（reference）」が分かれたプロジェクトです。貢献の入口も2系統あります。
小さく・根拠を添えて、が基本方針です。

## まず読むもの

- [規格本体 spec/ordito-spec.md](spec/ordito-spec.md) — MUST/SHOULD/MAY が要件の強さ。
- [docs/](docs/) の所見メモ — 「なぜこの仕様か」の設計判断の記録。提案前に既出の議論を確認できます。

## 何に貢献するか

| 対象 | 例 | 注意 |
|------|----|------|
| **規格（spec/）** | 語彙の追加、契約フィールドの規定、TBD の解決 | コアの安定が最優先。互換性への影響を必ず明記。 |
| **参照実装（reference/）** | エンジンのバグ修正、テンプレート追加、レンダラ改善 | 規格に準拠していること。`conformance/` が通ること。 |
| **準拠テスト（conformance/）** | 検証ケースの追加、スキーマの精緻化 | 実装非依存（入力IR＋期待出力）であること。 |
| **スキル（.claude/skills/）** | 新しい書き込み/読み出しスキル | ストアへの「書き込み」か「読み出し」のどちらかに収まること（§7）。非対話・冪等・JSON I/O。 |

## バージョニング（規格）

規格は **セマンティックバージョニング**（SemVer）に従います。

- **MAJOR**: IR / 契約 / コレクションのスキーマの**破壊的変更**（既存の準拠 IR が無効になる等）。
- **MINOR**: 後方互換な追加（新しいブロック型、任意フィールド、新スキル等）。
- **PATCH**: 記述の明確化・誤りの修正（挙動の変更なし）。

破壊的変更の提案は、移行の指針（旧→新の変換方法）を添えてください。

## Issue / PR の粒度

- **Issue 先行**: 規格の変更や新機能は、まず Issue で意図と影響範囲を合意してから PR を。
  バグ修正は Issue 無しの PR でも可。
- **PR は小さく**: 1 PR = 1 つの論点。規格変更と実装変更が絡む場合も、レビューしやすい単位に分割を。
- **根拠を添える**: 「なぜ必要か」を、できれば実例（失敗するサンプル等）とともに。Ordito の仕様は
  POC で実際にぶつかった穴を埋める形で育ってきました（docs/ の所見が証拠）。同じ姿勢を歓迎します。
- **準拠を壊さない**: 実装 PR は `node conformance/run.js` が green であること。

## 開発の前提

- Node.js v18+ のみ（参照実装は依存パッケージ・ビルド工程なし）。
- 秘密情報（API キー等）をコミットしない。AI 生成を使う機能は API キーを環境変数から読む設計に。
